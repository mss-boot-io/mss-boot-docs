## 概述

在四期、五期完成后，`mss-boot-admin` 的下一阶段不再以新增功能为主，而是转向：

- 把现有能力做顺
- 把高频问题做稳
- 把页面体验做统一
- 把代码结构做清晰
- 把可复用能力沉淀下来

> **阶段判断**：当前阶段更适合定义为“产品打磨与工程治理阶段”，而不是“继续扩功能阶段”。

## 阶段目标

本阶段目标不是扩展能力面，而是提升现有产品的：

1. **稳定性**：关键流程少出错、异常可控、边界清晰。
2. **一致性**：页面、交互、文案、接口行为、错误提示统一。
3. **可维护性**：职责边界更清晰，重复逻辑更少，后续开发不继续变乱。
4. **可复用性**：沉淀通用页面骨架、通用组件、通用后端处理模式。
5. **产品感**：页面视觉层次、加载态、空态、错误态、反馈体验更完整。

## 非目标

本阶段默认**不以以下事项为主线**：

- 不追求新增大型业务模块
- 不继续扩大功能清单
- 不回到代码生成/低代码主线
- 不重新引入多租户主线
- 不为了“看起来高级”而进行大规模无边界重构

## 阶段原则

### 1. 稳定性优先于功能数量

任何会影响登录、权限、上传、监控、任务、通知、日志等关键链路的问题，优先级高于新增功能。

### 2. 一致性优先于局部最优

单个页面“能跑”不等于合格。页面之间、模块之间、接口之间应尽量遵循同一套约定。

### 3. 小步整改，逐块验收

优先处理高收益、低风险、可验证的整改项，避免一次性大拆大改造成新不稳定。

### 4. 抽象必须来源于真实重复

只有在多个页面、多个模块已经出现重复模式时，才上升为通用组件或通用服务，避免“先设计一套很大抽象”导致复杂度反增。

### 5. 打磨必须可验证

每一项整改都应有明确的验收方式，例如：

- 是否消除了已知错误路径
- 是否统一了页面结构
- 是否减少了重复逻辑
- 是否补齐了加载态/空态/错误态
- 是否通过了最小必要验证

## 四条主线

## 1. 稳定性治理

重点解决：

- 高风险 panic / nil dereference / index 越界
- 鉴权覆盖不完整
- 重复关闭资源、定时器泄漏、轮询未清理
- 关键链路错误处理不一致
- API 返回格式与绑定方式不统一

重点模块：

- 登录与认证链路
- 菜单与权限链路
- 运行日志与监控链路
- 任务执行链路
- 告警与通知链路

### 验收口径

- P0 风险项全部清理
- 关键页面和关键接口不再存在明显错误路径
- 高优先级问题有最小回归验证

## 2. UI/UX 统一与页面美化

重点解决：

- 页面容器结构不统一
- 表格页、表单页、详情页缺少统一骨架
- 按钮、间距、状态色、说明文案风格不一致
- 加载态、空态、错误态缺失
- 同类页面交互行为不同

重点模块：

- Welcome / 监控页
- 登录 / 注册 / 找回密码 / 重置密码
- 日志页
- 系统配置 / 应用配置页
- 常见 CRUD 页面（用户、角色、部门、菜单、任务等）

### 验收口径

- 页面主骨架统一
- 关键页面具备明确加载态/空态/错误态
- 同类操作行为一致（新增、编辑、返回、保存、详情查看）

## 3. 前后端代码整治

重点解决：

- 控制器过重、模型副作用过重、服务层边界不清
- 通用分页、配置、日志、错误处理逻辑重复
- 前端页面骨架、配置表单、监控逻辑重复
- 命名不一致、目录职责漂移、遗留无效组件

### 验收口径

- 明显重复逻辑被收敛
- 新旧职责边界更清晰
- 目录和命名更容易理解
- 不引入新的并行实现方式

## 4. 通用能力沉淀

重点沉淀：

- 前端通用列表页/表单页/详情页模式
- 通用 AppConfig 子页模式
- 通用监控展示模块
- 后端统一绑定/校验/错误响应约定
- 审计/日志统一写入通道
- 配置存储抽象与驱动边界

### 验收口径

- 至少完成一轮“从重复实现到通用抽象”的闭环
- 新模块开发可直接复用既有模式，而不是重新发明一套

## 分阶段推进建议

## P1：稳定性基线治理

目标：先把最容易出事故、最影响信心的问题清掉。

建议内容：

- 修复 panic / nil / 越界风险
- 补齐敏感接口鉴权
- 清理未释放轮询与定时器
- 修正明显错误跳转与错误分支判断
- 统一关键错误反馈

## P2：页面骨架统一

目标：让主要页面看起来像同一个产品，而不是多个页面各写各的。

建议内容：

- 统一 PageContainer 使用方式
- 统一列表页 toolbar / drawer / form 布局
- 统一按钮布局与交互反馈
- 补齐 loading / empty / error 状态

## P3：代码收敛与复用抽象

目标：减少重复代码，避免后续继续写乱。

建议内容：

- 前端抽通用页面骨架与公共组件
- 后端收敛审计/日志/配置/错误处理模式
- 清理无效组件、无效页面、重复逻辑

## P4：质量基线制度化

目标：把“这次打磨”变成“后续不再持续变乱”的约束机制。

建议内容：

- 建立页面一致性检查清单
- 建立关键链路回归清单
- 建立新增页面/接口的通用模板要求
- 建立代码评审重点项

## 当前已知问题依据

本方案基于现有代码审计，已发现以下方向性问题：

- 前端页面存在大量重复页面骨架与内联样式
- 登录、重置密码、监控等关键页面存在稳定性与体验粗糙点
- AppConfig 与监控相关页面存在重复实现
- 后端存在若干 panic 风险、鉴权遗漏、错误处理不一致
- 审计日志与配置逻辑存在可收敛的重复实现

详细条目见：

- [首轮整改清单](/admin/product-polish-remediation-round-1)

## 角色分工建议

### leader

- 负责范围收敛、优先级和整改顺序
- 保证“不扩功能，只打磨”的阶段纪律

### backend developer

- 负责稳定性基线、鉴权修复、错误处理统一、通用服务收敛

### frontend developer

- 负责页面骨架统一、视觉一致性、交互一致性、公共组件抽象

### dev-test

- 负责关键链路回归清单、验收步骤、文档沉淀

## 完成标准

当以下条件同时满足时，可认为本阶段取得阶段性结果：

- P0 / P1 问题完成清理并验证
- 核心页面的视觉与交互风格明显统一
- 关键重复逻辑完成第一轮收敛
- 有清晰的通用模板/约定可供后续继续使用
- 回归与发布检查清单补齐并开始执行
