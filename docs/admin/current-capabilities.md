## 说明

本文基于当前仓库中可见的 `mss-boot-admin` 实现进行梳理，目标是帮助使用者快速理解当前产品已经具备的能力范围。

本文主要依据以下证据来源整理：

- `mss-boot-admin` README
- `mss-boot-admin/apis/*.go`
- `mss-boot-admin/models/*.go`
- `mss-boot-admin` 迁移种子中的默认菜单与模块分组

这是一份“当前实现能力”视角的总览，不是路线图，也不保证所有能力在任意部署中都默认开启。

## 功能分层

当前 `mss-boot-admin` 可以理解为一个以权限管理为基础、同时具备组织管理、系统配置、通知任务、国际化、监控统计，以及部分开发辅助能力的后台管理平台。

从使用视角看，现有能力大致可以分为五组。

## 1. 认证、账号与访问控制

这一层解决“谁可以进入系统、以什么身份工作、拥有哪些权限”。

当前可见能力包括：

- 用户登录、刷新 token、获取当前用户信息
- 用户资料维护与头像上传
- 用户主动修改密码，管理员重置用户密码
- GitHub / Lark 等第三方登录回调与绑定能力
- Personal Access Token 的生成、刷新、撤销与查询
- 基于 Casbin 的 RBAC 权限控制
- 菜单权限、API 权限、角色授权关系管理

这意味着系统不仅能完成基本登录，还提供了适合后台管理系统的权限治理与账号自服务能力。

## 2. 组织与基础资料管理

这一层解决“系统中有哪些人、部门、岗位，以及业务中通用的选项数据如何维护”。

当前可见能力包括：

- 用户管理
- 部门管理
- 岗位管理
- 角色管理
- 菜单管理
- API 注册与维护
- 选项管理（用于动态枚举/字典项维护）

这些能力构成了后台系统最常见的组织与主数据基础层，也为权限与页面配置提供了承载。

## 3. 系统配置、通知与运行能力

这一层解决“系统如何按环境配置、如何通知用户、如何查看部分运行状态”。

当前可见能力包括：

- 系统配置管理
- 应用配置与用户配置相关接口
- 通知公告与未读/已读处理
- 定时任务管理及执行日志相关能力
- 监控信息接口（如 CPU、内存、磁盘等）
- 统计查询接口

这一部分说明 `mss-boot-admin` 不只是“做 CRUD 的后台”，它也在向可运营、可维护的平台能力延伸。

## 4. 国际化与内容适配

当前仓库中可以看到系统已经具备国际化相关能力，包括：

- 国际化资源管理
- 多语言相关接口与数据模型
- 与页面配置、菜单、动态模型协同的语言资源支持

这使它更适合做面向多语言场景的管理系统，而不是仅限单语言后台。

## 5. 开发辅助能力（当前仍存在，但已不再是主线）

这组能力在当前仓库中仍然存在，但从产品方向上看，后续不再建议把它作为核心主线继续强化。

当前可见能力包括：

- 虚拟模型管理
- 模型字段管理
- 动态生成数据相关能力
- 模型文档接口
- 模板管理与代码生成能力
- 文件上传接口
- WebSocket 事件接口

从仓库实现看，系统并不只关注“后台页面管理”，也具备一定的平台扩展与开发辅助基础。不过在当前阶段，这些能力更适合被视为历史存量能力，而不是后续主叙事。

## 当前内置功能可以怎样理解

如果从产品菜单角度看，当前能力大致可以理解为以下几类模块：

- **权限与身份**：登录、角色、菜单、API、Token、OAuth2
- **组织与用户**：用户、部门、岗位
- **系统与运营**：系统配置、通知公告、任务管理、监控、统计
- **内容与国际化**：选项管理、国际化资源
- **平台与扩展**：上传、事件、以及部分历史开发辅助能力

## 使用这份总览时需要注意的边界

### 1. 部分能力依赖运行时配置

例如 OAuth2、邮件、对象存储等能力，在代码和接口层已实现，但是否可直接使用仍依赖部署时配置。

### 2. 菜单可见性可能因初始化数据不同而变化

本文参考了当前仓库中的迁移种子与菜单分组；不同部署若调整了初始化数据，实际后台可见菜单可能有所差异。

### 3. 多租户能力在代码层存在，但管理入口仍需结合实际部署判断

从模型基础类型和部分实现可以看出多租户设计已经存在，但具体“租户管理”作为独立产品模块的暴露方式，仍建议结合实际部署菜单与配置确认。
