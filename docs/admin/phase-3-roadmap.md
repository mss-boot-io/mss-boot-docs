## 说明

本文对应 `mss-boot-admin` 在完成一期“产品方向统一”和二期“AI 注解协同规范”之后的下一阶段工作重点。

三期的目标不是继续扩大功能面，而是围绕已经明确的产品主线，补强真正决定平台长期价值的能力。

当前阶段的三期主线可以概括为：

> 从“方向统一”进入“能力补强”，围绕治理、运营、协作和扩展四个方向，形成更稳的后台平台基础。

## 三期总目标

三期希望解决的核心问题是：

- 当前能力虽然已经不算少，但在产品表达上仍然偏“模块集合”，而不是“能力体系”。
- 权限、组织、配置、通知、任务、监控、统计这些能力已经存在，但仍需要按平台方式重新组织与强化。
- AI 注解协同规范已经建立，但还需要与具体能力建设路线结合，而不是停留在独立文档层。

因此，三期的目标不是“再发明新卖点”，而是让现有能力形成稳定的治理与运营骨架。

## 路线图结构

三期建议按五条能力线推进，并保持优先级清晰。

## P1. 治理基线补强（最高优先级）

这是三期最核心的能力线。

目标：把当前已经存在的权限与组织相关能力，从“有模块”提升为“有边界、有规则、有协同逻辑的治理体系”。

优先覆盖范围：

- 用户、角色、部门、岗位
- 菜单与 API 权限
- OAuth2、PAT、访问边界
- 配置权限与环境隔离

这一条线应重点回答：

- 谁能访问系统
- 谁能操作哪些资源
- 不同组织/角色的边界如何表达
- 配置、接口、菜单和权限如何保持一致

预期结果：

- 平台对“治理核心”有统一表述
- 角色、权限、组织、菜单、API 的关系更容易理解与维护
- 后续新模块接入时不再绕开现有治理模型

## P2. 运营中枢补强

这一条线解决“系统上线后如何运转、维护和响应日常运营需求”。

优先覆盖范围：

- 系统配置
- 通知公告
- 定时任务
- 监控信息
- 统计查询

重点不是简单保留这些模块，而是把它们组织成一个更完整的运营控制面。

这一条线应重点回答：

- 平台如何支持日常运行维护
- 通知、任务、监控、统计之间怎样形成闭环
- 配置变更后如何被验证、感知和追踪

预期结果：

- `mss-boot-admin` 的“运营能力”可以被单独作为一条产品能力线讲清楚
- 文档不再只是列模块，而是能解释模块之间的配合关系

## P3. AI 注解协同流程落地

二期已经定义了 AI 注解协同规范，三期要让它真正进入能力建设流程。

优先覆盖范围：

- 治理类评审文档
- 运营能力规划文档
- 前后端联调与交接文档
- dev-test 的验证与沉淀输出

这一条线不是新增“功能模块”，而是把 AI 注解从抽象规范变成流程基础设施。

这一条线应重点回答：

- 哪些治理/运营模块改动必须配套注解文档
- leader 怎样基于注解把任务派给 architect / backend / frontend / dev-test
- 如何通过评审、计划、交接、评估四类产物保证多角色一致性

预期结果：

- 重要改动不再只存在于会话或个人记忆中
- 团队对“怎么写交接、怎么做评审、怎么形成下一步”有统一模板和共识

## P4. 集成与扩展护栏

这条线用于保留平台的扩展能力，但防止扩展绕开治理与运营主线。

优先覆盖范围：

- 国际化
- 对象存储 / 上传
- WebSocket 事件能力
- API-first 扩展方式

这一条线应重点回答：

- 新能力如何接入现有治理与运营体系
- 扩展模块应遵守什么边界
- 哪些扩展属于平台能力，哪些属于业务层能力

预期结果：

- 扩展能力继续保留，但不会重新把产品叙事带回“功能散点化”
- 新能力接入更容易被评审、被验证、被文档化

## P5. 历史能力降级与叙事收口

这一条线不是主建设线，但它非常重要，因为它决定产品对外口径是否一致。

优先覆盖范围：

- 动态模型
- 模板 / 代码生成
- 历史“开发辅助能力”的对外表述

重点不是立刻删除，而是控制其对外叙事与后续投资预期。

这一条线应重点回答：

- 哪些历史能力继续保留为存量能力
- 哪些能力进入冻结或弱化状态
- 文档中如何表达“仍存在”与“非主线”的关系

预期结果：

- 产品主线不再被旧叙事反复打断
- 团队和外部使用者都能理解当前重点到底是什么

## 推荐推进顺序

建议按以下顺序推进三期：

1. **P1 治理基线补强**
2. **P2 运营中枢补强**
3. **P3 AI 注解协同流程落地**
4. **P4 集成与扩展护栏**
5. **P5 历史能力降级与叙事收口**

原因很简单：

- 治理与运营决定产品基础是否稳
- AI 注解流程要服务具体能力建设，而不是空转
- 历史能力收口需要建立在新主线已经足够清晰之后

## 三期完成后应达到的状态

如果三期推进有效，平台应逐步呈现以下结果：

- 用户、角色、组织、菜单、API、配置等治理能力有统一表达
- 通知、任务、监控、统计被组织成更完整的运营能力
- AI 注解成为多角色协作的标准基础设施
- 国际化、上传、事件等扩展能力有清晰接入边界
- 动态模型与代码生成被稳定降级为历史能力，而不再影响主叙事

## 当前阶段建议

如果三期要继续拆得更细，建议下一步优先把以下三篇专题文档补出来：

1. **权限与组织治理说明**
2. **运营能力说明（配置 / 通知 / 任务 / 监控 / 统计）**
3. **Token / OAuth2 / 联调与审计说明**

这三篇会直接决定三期是否能从“路线图”进入“模块级能力整理”。
