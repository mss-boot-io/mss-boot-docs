## 目标

本规范用于定义 `mss-boot-admin` 当前阶段的 **AI 注解协同方式**。

这里的“AI 注解”不是指一套自动生成代码的黑盒机制，而是指：

- 用于表达设计意图的结构化说明
- 用于约束实现边界的上下文说明
- 用于支撑文档、交接、评审、回归的统一依据

换句话说，AI 注解是 **协作契约**，而不是“替开发做决定”的魔法功能。

## 适用范围

本规范适用于以下协作场景：

- 产品方向讨论
- 架构评审
- 需求拆解
- 实施计划
- 前后端联调
- 测试回归
- 交接与文档沉淀

本规范主要作用于：

- `mss-boot`
- `mss-boot-admin`
- `mss-boot-admin-antd`
- `mss-boot-docs`

## 为什么要做这件事

在当前产品方向下，`mss-boot-admin` 不再把动态模型与代码生成作为主线能力继续强化。

取而代之的是：

- 用更清晰的治理和运营能力定义产品主线
- 用 AI 注解提高设计、实现、文档、交接之间的一致性
- 用可验证的过程降低“靠口头同步”和“靠个人记忆协作”的成本

因此，AI 注解的核心价值不是替代工程，而是让工程更容易被团队协作、评审、交接和维护。

## AI 注解的定义

在本项目中，AI 注解至少应承担以下职责：

1. **表达目标**：说明这次变更是为了解决什么问题。
2. **表达边界**：说明做什么、不做什么、影响到哪些模块。
3. **表达约束**：说明兼容性、风险、依赖、验收边界。
4. **表达证据**：说明结论来自哪些代码、文档、接口或运行事实。
5. **表达交接信息**：说明谁可以继续接手、如何验证、还有什么风险。

如果某段注解不能支撑以上至少两项职责，它通常就不值得存在。

## 不要把 AI 注解理解成什么

### 1. 不是低代码配置替代物

AI 注解不负责在运行时动态生成模型、页面或模块。

### 2. 不是自动生成一切的承诺

它不能替代架构判断、实现质量控制、测试验证和交付责任。

### 3. 不是冗长自然语言堆砌

注解必须服务协作，不能只是把人人都知道的话换一种方式重复一遍。

### 4. 不是“写给 AI 看，团队看不懂”的特殊语言

注解必须同时对人类协作者和 AI 协作者可理解。

## 核心原则

## 1. 基于事实

- 只基于当前仓库可见代码、文档和已知运行事实写结论。
- 必要时明确适用范围、基线时间和假设条件。

## 2. 面向协作

- 注解要帮助下一个角色继续工作。
- 不写“只有作者本人看得懂”的局部上下文。

## 3. 以最小充分信息为目标

- 信息必须足够支撑理解与接手。
- 但不追求把所有背景都塞进一个文档。

## 4. 结论可验证

- 所有关键判断都应能被代码、文档、接口、命令、页面路径或测试方式复核。

## 5. 面向产品主线

- 注解优先服务治理、运营、AI 协同、集成扩展这四类长期能力。
- 不再围绕动态模型与代码生成去组织新的主叙事。

## 哪些场景必须产出 AI 注解

以下场景默认应产出结构化注解文档，而不是只停留在会话中：

### 1. 架构评审

例如：

- 控制器 / Action / Hook 机制演进
- 权限模型、Scope、鉴权链路调整
- 多项目共用能力边界调整

### 2. 中大型需求改造

例如：

- 去多租户
- 配置体系调整
- 登录方式调整
- 组织或权限模型改造

### 3. 跨角色交接

例如：

- leader 派发到 backend / frontend / dev-test
- backend 与 frontend 的联调交接
- dev-test 向 docs 沉淀验证结果

### 4. 需要复盘或复用的决策

例如：

- 为什么保留某个架构抽象
- 为什么下线某个旧能力
- 为什么某条路线暂缓而不是立刻执行

## 注解产物类型

结合仓库中现有高质量样例，当前建议保留四类主产物。

## 1. 评审类（Review）

适用于：架构评估、能力优劣分析、方向判断。

典型结构：

- 适用范围 / 基线
- 结论速览
- 当前设计或实现梳理
- 优势与代价
- 风险与限制
- 建议路线

现有参考：

- `action-architecture-review.zh-CN.md`
- `action-architecture-one-pager.zh-CN.md`

## 2. 计划类（Plan）

适用于：需求改造、阶段拆解、PR 切分。

典型结构：

- 背景与目标
- 影响范围
- 风险分级
- 分阶段任务
- 验收标准
- PR 切分建议

现有参考：

- `tenant-removal-impact-plan.zh-CN.md`
- `frontend-tenant-removal-impact-plan.zh-CN.md`

## 3. 交接类（Handoff）

适用于：中途暂停、角色接手、阶段收尾。

典型结构：

- 已完成结论
- 影响范围摘要
- 风险分级
- 下一步建议顺序
- 启动前检查项
- 验证命令或回归清单
- 相关参考文档

现有参考：

- `tenant-removal-handoff-summary.zh-CN.md`
- `frontend-tenant-removal-handoff-summary.zh-CN.md`

## 4. 评估类（Evaluation）

适用于：某个主题的现状评估、收益/代价比较、路线预判。

典型结构：

- 评估时间与范围
- 证据来源
- 当前实现观察
- 优势 / 问题 / 成本
- 建议结论
- 可复现验证方式

现有参考：

- `multi-tenant-cache-evaluation.zh-CN.md`
- `frontend-multi-tenant-cache-evaluation.zh-CN.md`

## 每份 AI 注解至少要回答的 6 个问题

1. 这份文档要解决什么问题？
2. 它基于哪些可见证据？
3. 结论适用于什么范围，不适用于什么范围？
4. 有哪些主要风险和未决事项？
5. 下一个角色拿到它后应该做什么？
6. 怎么验证这份结论没有脱离现实？

## 与角色协作的关系

AI 注解并不是脱离角色体系存在的，它应当配合 leader-first 协作模型工作。

### leader

- 决定是否需要产出注解
- 决定是哪一类注解
- 决定何时回收并转交给下一角色

### architect

- 负责边界、权衡、阶段路线类注解

### backend / frontend

- 负责实现相关的契约、影响面、验证方式和联调说明

### dev-test

- 负责把验证结论、测试证据、文档沉淀要求补齐

## 命名与落盘原则

- 项目内提示词与分析文档继续遵循各项目 `aigc/prompts/` 约束。
- 面向产品使用者和开源协作者的长期文档，优先沉淀到 `mss-boot-docs`。
- 文件名使用小写 kebab-case，中文文档使用 `.zh-CN.md` 后缀。

## 质量红线

以下内容不应出现在 AI 注解中：

- 没有证据支持的绝对结论
- 只写结论，不写适用范围和边界
- 用内部黑话代替具体模块、路径、接口或页面
- 没有下一步建议的交接文档
- 没有验证方式的评估文档
- 明知与当前产品主线冲突却继续强化旧叙事

## 二期完成定义

当以下条件成立时，可认为 AI 注解协同规范进入可用状态：

1. 团队知道哪些场景必须写注解。
2. 团队知道评审、计划、交接、评估四类文档的最小结构。
3. leader 能用这套规范把需求推进到下一个角色。
4. dev-test 能把结果沉淀成对外可复用文档。
5. 新文档不再围绕动态模型/代码生成展开主叙事。
