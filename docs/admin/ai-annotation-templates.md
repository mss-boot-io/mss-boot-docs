## 说明

本文不是要求所有文档都长得完全一样，而是给出当前阶段推荐的最小结构。

目标是让不同角色写出来的 AI 注解，能够被后续角色稳定接住。

## 1. 评审类模板

适用场景：

- 架构评审
- 方案评估
- 方向判断

推荐结构：

```md
# 标题

> 适用范围：基于当前仓库可见代码与文档（截至 YYYY-MM-DD）

## 1. 结论速览
- 关键结论 1
- 关键结论 2

## 2. 当前实现/设计梳理
- 事实 1
- 事实 2

## 3. 优势
- ...

## 4. 风险与代价
- ...

## 5. 建议路线
- 阶段 A
- 阶段 B

## 6. 推荐结论
- ...
```

## 2. 计划类模板

适用场景：

- 需求改造
- 中大型重构
- 跨模块实施拆解

推荐结构：

```md
# 标题

> 背景：...
> 范围：...

## 1. 目标
- ...

## 2. 影响范围
- 模块/目录/接口

## 3. 风险分级
### 高风险
- ...

### 中风险
- ...

### 低风险
- ...

## 4. 分阶段任务
### Phase 1
- ...

### Phase 2
- ...

## 5. 验收标准
- ...

## 6. PR 切分建议
- PR-1
- PR-2
```

## 3. 交接类模板

适用场景：

- 中途暂停
- 角色接手
- 阶段收尾

推荐结构：

```md
# 标题

> 适用范围：...

## 1. 已完成结论
- ...

## 2. 当前影响范围
- ...

## 3. 风险分级
- ...

## 4. 下一步建议顺序
1. ...
2. ...

## 5. 启动前检查项
- ...

## 6. 验证/回归清单
- ...

## 7. 相关参考文档
- ...
```

## 4. 评估类模板

适用场景：

- 现状评估
- 能力对比
- 成本/收益分析

推荐结构：

```md
# 标题

> 评估时间：YYYY-MM-DD
> 评估范围：...

## 1. 证据来源
- ...

## 2. 当前观察
- ...

## 3. 优势
- ...

## 4. 问题/限制
- ...

## 5. 建议结论
- ...

## 6. 可复现验证方式
- ...
```

## 5. 交付时最少要补充的内容

无论采用哪种模板，最终至少补齐以下信息：

- 适用范围
- 关键结论
- 风险与未决事项
- 下一步建议
- 验证方式或验证入口

## 6. 推荐使用顺序

在 leader-first 协作模型下，推荐这样使用：

1. 先写评估或评审，确认方向
2. 再写计划，形成实施顺序
3. 实施中持续补交接文档
4. 收尾时沉淀成 docs 可复用内容
