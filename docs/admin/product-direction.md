## 背景

结合当前仓库实现、已有文档，以及与 HotGo 等开源后台项目的能力对比，可以看到：

- 权限、组织、菜单、日志、任务、代码生成、表单生成，已经是 Go admin 赛道的常见竞争区。
- `mss-boot-admin` 当前已经具备权限治理、组织管理、系统配置、通知任务、国际化、监控统计等较完整的后台基础能力。
- 动态模型与代码生成虽然在仓库中已有实现基础，但目前并不是最适合继续投入的主方向。

因此，当前阶段更合适的产品策略不是继续扩张功能面，而是先收敛产品主线。

## 当前方向判断

`mss-boot-admin` 后续更适合被定义为：

> 一个面向真实后台系统的 **治理型 + 运营型 + AI 注解协同型 admin 平台**。

这意味着产品重点不再放在“更快生成 CRUD”或“更强低代码”，而是放在：

- 更清晰的权限与组织治理
- 更稳定的系统配置与运营能力
- 更实用的通知、任务、监控、统计能力
- 更适合团队协作的 AI 注解驱动研发流程

## 不再作为主线投入的方向

当前阶段，以下能力不再建议作为产品主线继续强化：

- 动态模型
- 代码生成 / 微服务代码生成

说明：

- 这并不等于当前仓库中已有实现立刻删除。
- 更准确的表述是：**进入弱化/降级阶段，不再作为核心卖点与主要演进方向。**
- 若后续完全退出，需要在实现、文档、迁移说明和用户沟通上分阶段完成。

## 替代主线是什么

放弃把“动态模型/代码生成”做成主线之后，平台需要强化以下五个能力支柱。

## 1. 治理核心

围绕后台系统最基础也最长期的能力继续做深：

- 用户、角色、部门、岗位
- 菜单与 API 权限
- PAT、OAuth2、访问边界
- 配置治理与环境隔离

## 2. 运营能力

围绕后台日常运营与维护场景继续强化：

- 通知公告
- 定时任务
- 监控信息
- 统计查询
- 配置变更与运行态协作

## 3. 集成与扩展能力

保留并强化已经有基础的扩展方向：

- 国际化
- 对象存储/上传
- WebSocket 事件能力
- API-first 的模块扩展方式

## 4. AI 注解驱动研发协作

这里的重点不是“AI 自动生成一切”，而是把 AI 注解用作工程协作契约，例如：

- 设计意图说明
- 接口语义说明
- 约束与边界说明
- 文档与交接输出依据
- 评审与回归时的上下文来源

## 5. 面向生产演进的产品表达

与其强调“低代码”“一键生成”，更适合强调：

- 长期可维护
- 权限治理清晰
- 运维与运营能力完整
- 多角色协作成本低
- AI 参与但不失控

## 一期目标（开发前阶段）

一期不做大规模实现调整，先完成方向统一。

当前阶段的一期目标是：

1. **统一口径**：README、文档首页、功能清单不再把动态模型/代码生成作为核心卖点。
2. **保留事实**：承认当前仓库里仍存在这些能力，但明确它们不再是重点演进方向。
3. **建立新主线**：对外统一为“治理型 + 运营型 + AI 注解协同型 admin 平台”。
4. **形成后续开发边界**：后续新增能力优先服务新的五个能力支柱，而不是继续扩大生成类能力面。

## 对比 HotGo 等项目时的建议表述

不建议继续从“谁生成更快”来描述差异。

更适合的对比方式是：

- HotGo 等项目偏向 **快速搭建、插件扩展、生成效率**
- `mss-boot-admin` 后续偏向 **治理清晰、运营稳定、长期演进、AI 协同研发**

## 之后的阶段建议

当一期完成后，下一步建议依次进入：

1. **文档与 README 全面清理旧叙事**
2. **明确 AI 注解契约与使用规范**
3. **围绕治理与运营能力补强具体模块**
4. **再决定是否对动态模型/代码生成做冻结、迁移或下线**
